This document explains how to set up a multinode Vespa system. Readers are expected to know how to set up a single-node system. Installing Vespa (on each node), deploying applications, feeding data and searching is done the same way as described in the quick-start for single-node instances.

Before setting a system into production, it is a good idea to do sizing and benchmarking exercise to detemine how many nodes you need to meet capacity. For production systems the smallest size of any cluster should be 2 for redundancy. The same nodes may run multiple clusters at the same time.

Installing and Running Step-by-Step

  • Install:

    Install Vespa on each node exactly as you would for a single-node instance. Even though the nodes may have different roles, they all have exactly the same software packages.

  • Create a multinode application: We'll extend the example application in $VESPA_HOME/share/vespa/sampleapps/search/music to specify multiple nodes. 

    $ cd [your working directory]
$ cp -r $VESPA_HOME/share/vespa/sampleapps/search/music music-multinode

    List all the host that should be part of the system in music-multinode/hosts.xml. For example 

    <?xml version="1.0" encoding="utf-8"?>
<hosts>
  <host name="node1.host.com">
    <alias>config0</alias>
  </host>
  <host name="node2.host.com">
    <alias>container0</alias>
  </host>
  <host name="node3.host.com">
    <alias>content0</alias>
  </host>
  <host name="node4.host.com">
    <alias>content1</alias>
  </host>
</hosts>

    The hosts.xml file simply provides an alias for each hostname such that the services file can refer to hosts by alias. The aliases are just ids and carry no meaning.

    Now, modify music-multinode/services.xml as follows: 
    <?xml version="1.0" encoding="utf-8" ?>
<services version="1.0">

  <admin version="2.0">
    <adminserver hostalias="config0" />
    <slobroks>
      <slobrok hostalias="config0"/>
      <slobrok hostalias="container0"/>
    </slobroks>
  </admin>

  <jdisc version="1.0">
    <search />
    <nodes>
      <node hostalias="container0 />
    </nodes>
  </jdisc>

  <content id="music" version="1.0">
    <redundancy>2</redundancy>
    <documents>
      <document type="music" mode="index" />
    </documents>
    <nodes>
      <node hostalias="content0" distribution-key="0"/>
      <node hostalias="content1" distribution-key="1"/>
    </nodes>
  </content>

</services>

    We have replaced all node1 references with the appropriate node alias for the new multinode layout. We also added a new location broker node for redundancy. For content, we have changed the redundancy (number of copies of each document) from 1 to 2, and added a new content node. Notice that it is possible to run any combination of services on the same node by adding the same alias multiple places. It is also possible to make changes to the set of nodes and redeploying while the system is serving queries and receiving writes. When you make changes to content clusters the system will gradually redistribute the stored data to the new nodes.

    Explanations of the different parts of services.xml are found in the application package reference doc. The Recommended hardware configuration doc gives some recommendations on the service distributions in multinode setups.

  • Deploy application: Now we are ready to deploy the multinode application, whcih is done once on any of the config server nodes, precisely the same way as for a single node system: 

    $ deploy prepare music-multinode
$ deploy activate

    The config server you deploy to will copy the application to any other config servers for redundancy and determine the detailed configuration to send to each of the other nodes. Whenever you want to change the application package, just do so and redeploy in exactly the same way. The config servers will figure out the delta and push it to the other nodes of the system, which in most cases will apply them safely while live. If the change cannot be safely done while the node is running it will hold off applying it until it is restarted. In cases where a restart is needed you'll be notified in the response from running deploy. It is also possible to deploy remotely over a web service API.

    With the deploy step the application should be up and running and you can feed data and query it as described e.g in the quick start. You can use the logfmt command to verify that there are no ERRORS and WARNINGS in the log. The log on config servers contains the messages from all the nodes in the system while other nodes only contain their own locally produced messages.
