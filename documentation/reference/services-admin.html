This is the reference documentation for the <admin> section in services.xml.

<admin version='4.0'>

Contained in: <services>
Admin version 4 is used for explicit control over the number of admin services running and whether these should run on dedicated nodes or on some existing container cluster in the application. In most cases, there is no need to specify any of this.

Sub elements

ElementQuantityDescription
<slobroks>/<nodes>0 or 1Provides control over nodes used for instance internal service location brokering.
<logservers>/<nodes>0 or 1Provides control over the node used as log server. At most 1 node can be configured.

<nodes>

<admin version='3.0'>

The content of admin version 3 is ignored. If this element is used, the result is the same as not having an admin element at all: The cluster will get a default setup of admin services on the same nodes used for other purposes (jdisc containers for slobroks and logserver and content nodes for cluster controllers). This is the perfect setup in most cases.

<admin version='2.0'> and <admin version='1.0'>

Admin version 2.0 provides the ability to set up config servers as well as the finer control over the administration services. The content of version 1.0 of this element is also documented below.

<cluster-controller>

Contained in: <cluster-controllers>
Valid Values: -
Default Value: -
Since: -
Deprecated: -
Specifies a host on which to run the Cluster Controller service. The Cluster Controller manages the state of the cluster in order to provide elasticity and failure detection.

Attributes

Common attributes: hostalias, baseport, jvmargs

Sub Elements

None.

Examples

<cluster-controllers>
  <cluster-controller hostalias="node1" />
</cluster-controllers>

<cluster-controllers>

Contained in: <admin>
Valid Values: -
Default Value: -
Since: -
Deprecated: -

This is a container for one or more <cluster-controller> elements.

Note: If your services.xml contains one or more <content> elements, configuring at least one Cluster Controller is required.

Attributes

  • standalone-zookeeper

    Valid Values:true/false
    Default Value:false
    Required:No, will by default share the ZooKeeper instance with configserver
    Since:-
    Deprecated:-
    If configured to true a separate ZooKeeper instance will be configured and started on the set of nodes where you run cluster controller on. The set of cluster controllers nodes cannot overlap with the set of nodes where config server is running. If this setting is changed from false to true in a running system all previous cluster state information will be lost as the underlaying ZooKeper changes. Cluster controllers will re-discover the state, but nodes that have been manually set as down will again be considered to be up.

Sub Elements

Element Required Quantity
<cluster-controller> Yes One

Examples

<cluster-controllers>
  <cluster-controller hostalias="node1" />
</cluster-controllers>
