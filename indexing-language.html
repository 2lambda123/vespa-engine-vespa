The indexing language is a part of the search-definition which defines how a field is processed during indexing.

In simple mode you specify the operations to be performed separated by the pipe (|) character, and leave the details to Vespa. If you have a advanced processing needs, you can either use advanced mode indexing language, or you can write a document processor.

Supported expressions in simple mode are:

attribute

Attribute is used to make a field available for sorting and grouping.

Attributes can be modified by the attribute: command in a search definition.

All data stored in attribute will be kept in memory at all times. It is therefore very important to avoid storing too much data in attributes, see the Attribute Sizing Guide for details.
index

Creates a searchable index for the values of this field. All strings are lower-cased before they are stored in the index.

By default the index name will be the same as the name of the search definition field. Optionally a field set can be specified to combine several fields in the same set for searching.

set_language

Parses the input value as an RFC 3066 language tag, and sets that language for the current document. This affects the behavior of the tokenizer. The recommended use is to have one field in the document containing the language code, and that field should be the first field in your document.

In a search definition, this operation must be placed before any indexed fields, as it will only affect the fields defined after it in the search-definition. If you have both header and body fields, the set_language-field should be header, if you require it to affect subsequent header fields and/or any body fields. The reason is the fields are iterated header-body when deriving the backend configuration for indexing scripts.

Please see the linguistics article for more information on how language settings are applied.

summary

Includes the value of this field in the document summary, i.e. the data returned to the user.

Summary output format can be modified by using the summary: command in a search definition (e.g. to make dynamic teasers).

Summary fields of type string are currently limited to 64kB. If a larger string is stored, the indexer will issue a warning and truncate the value to 64kB.
